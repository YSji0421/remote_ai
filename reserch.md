# AI 원격 상담 솔루션 웹 플랫폼 상세 분석 및 설계 보고서

## 1. 시스템 동작 흐름 (사용자 여정)
1. **인증 및 접근**: 상담사와 내담자는 이메일로 가입/로그인합니다. 상담사가 대시보드에서 '상담방 생성'을 클릭하면 고유한 방 입장 링크(또는 코드)가 생성됩니다.
2. **연결 (WebRTC)**: 내담자가 링크를 통해 접속하면, 대기 화면을 거쳐 양측의 카메라/마이크 접근 권한을 확인한 후 P2P 화상 통화가 연결됩니다.
3. **실시간 통화 및 STT**: 통화가 진행되는 동안, 각 사용자의 음성은 서버로 스트리밍되어 `faster-whisper` 모델을 통해 즉시 텍스트로 변환됩니다. 변환된 텍스트는 누가 말했는지(상담사/내담자) 타임스탬프와 함께 화면 하단 자막으로 실시간 전송되며 DB에 저장됩니다.
4. **상담 종료 및 AI 요약**: 통화가 종료되면 모아진 전체 STT 텍스트가 Gemini API로 전송됩니다. Gemini는 프롬프트 지시에 따라 '주요 내용', '내담자 호소 내용', '상담사 개입 내용'으로 구조화된 요약 리포트를 반환합니다.
5. **사후 관리**: 대시보드에서 과거 상담 기록, STT 원본, AI 요약본을 열람하고 PDF로 다운로드할 수 있습니다.

## 2. 권장 기술 스택 및 아키텍처
* **프론트엔드**: Next.js (React 기반, 반응형 웹, 초기 렌더링 및 SEO/성능 우수) + Tailwind CSS
* **백엔드**: Node.js (NestJS) + Python (FastAPI)
  - Node.js: 비즈니스 로직(회원가입, 방 생성, DB 저장, Gemini API 연동) 및 시그널링 서버
  - Python: `faster-whisper`를 활용한 실시간 STT 전용 AI 마이크로서비스
* **데이터베이스**: PostgreSQL (회원 및 상담 데이터) + Redis (세션 예약 및 실시간 상태 관리)
* **인프라/통신**: LiveKit (WebRTC SFU 서버) 또는 PeerJS (순수 P2P)

## 3. 🚨 잠재적 리스크 및 전문가 제안 (Proposals)

### ① WebRTC P2P 방식과 서버형 실시간 STT의 충돌 (구조적 모순)
* **이슈**: WebRTC P2P는 영상/음성을 사용자끼리 직접 교환하므로 서버를 거치지 않습니다. 하지만 STT 처리는 Python 서버에서 이루어져야 합니다. 프론트엔드에서 음성을 서버로 별도 전송(WebSocket 등)하면 네트워크 트래픽이 2배로 늘고 설계가 복잡해집니다.
* **대안 제안**: 순수 P2P 대신 **SFU(Selective Forwarding Unit) 방식의 WebRTC 미디어 서버(예: LiveKit) 도입**을 강력히 권장합니다. 서버가 미디어를 중계하므로, 백엔드에서 오디오 트랙을 추출해 `faster-whisper` 모델로 던져주기 훨씬 수월해지며, 3인 이상의 다중 상담으로도 쉽게 확장할 수 있습니다.

### ② faster-whisper의 실시간(Streaming) 처리 본질적 한계
* **이슈**: Whisper 모델은 본래 완성된 오디오 파일을 녹음 후 일괄 변환하는 데 특화되어 있어, 실시간(초단위) 스트리밍 처리를 위해서는 추가적인 트릭이 필요합니다.
* **대안 제안**: Python FastAPI 서버 내에 VAD(Voice Activity Detection - 짧은 침묵 감지) 라이브러리를 적용하여, 오디오 스트림을 문장 형태의 단위(Chunk)로 잘라서 Whisper에 비동기로 넘기는 파이프라인(퍼즐 맞추기 방식)을 구축해야 끊김없는 STT가 가능해집니다.

### ③ 화자 분리(Speaker Diarization) 로직의 최적화
* **이슈**: 여러 사람이 한 마이크에 대고 말할 때 누구의 목소리인지 구분하는 기술(Diarization)은 매우 무겁고 오류가 잦습니다.
* **해결책**: 원격 화상은 각 사용자의 오디오 채널(스트림)이 완전히 독립되어 있습니다. 굳이 AI로 화자를 분리할 필요 없이, **각 오디오 채널별로 별도의 STT를 실행하고 텍스트에 화자 식별자(상담사/내담자)를 명시하여 타임스탬프 순으로 병합**하는 방식을 채택하면 100%의 정확도로 발화자를 구분할 수 있습니다.

### ④ 민감한 의료/상담 정보 보안
* **이슈**: 내담자의 사적인 상담 기록이 DB에 평문으로 저장되면 내부자 유출이나 해킹 시 법적인 치명타를 입을 수 있습니다.
* **대안 제안**: 회원 정보의 비밀번호 암호화는 기본이며, STT 기록과 AI 요약 텍스트 전체를 **DB 저장 단계에서 AES-256 등 양방향 암호화 처리**하고, 권한이 있는 사용자(본인 및 담당 상담사)가 열람을 요청할 때만 복호화하여 전달하도록 설계해야 합니다.

## 4. 확장성 및 관리자 기능 청사진
* **관리자 (Admin)**: 전체 회원 및 개설된 방 현황을 모니터링할 수 있는 별도의 통합 대시보드 구축.
* **분석 고도화**: 추후 '자살, 자해' 등 위험 키워드가 STT에서 도출 시 상담사 화면에 Alert를 띄워주는 **실시간 위험 탐지 시스템**으로 확장이 용이한 마이크로서비스 구조 채택.
